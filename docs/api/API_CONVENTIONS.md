# API Conventions

**Status:** Living contract; authentication, health, and chat endpoints implemented
**Last updated:** 2026-07-13  
**Base path:** `/api/v1`  
**Related:** [Data model](../architecture/DATA_MODEL.md), [Threat model](../security/THREAT_MODEL.md)

## 1. Principles

- JSON over HTTPS; UTF-8; resource-oriented paths.
- Pydantic v2 schemas validate all request, response, worker, and provider boundaries.
- Routes handle HTTP translation; services own authorization and use cases; repositories own persistence.
- Backwards-compatible additions are preferred within `v1`; breaking contract changes require a new API version and migration plan.
- OpenAPI generated by FastAPI is the contract source for generated TypeScript clients.

## 2. Naming and representation

- Paths use plural kebab-case nouns: `/enterprise-profiles/{profile_id}/assessments`.
- JSON fields use `snake_case` consistently across Python and generated TypeScript types.
- IDs are opaque UUID strings. Clients must not infer meaning from them.
- Timestamps use RFC 3339 UTC, for example `2026-07-13T10:30:00Z`.
- Calendar dates use `YYYY-MM-DD`; monetary values use decimal strings plus ISO 4217 currency.
- Optional and nullable are distinct. Omitted means not supplied; `null` is accepted only when the schema explicitly permits clearing/unknown.
- Enumerations are lowercase snake case and documented as closed unless explicitly marked extensible.
- Responses do not expose database models directly.

## 3. Authentication and authorization

- The MVP uses the first-party identity and encrypted-profile design in [ADR-0006](../architecture/adr/0006-first-party-identity-and-encrypted-profiles.md): Argon2id passwords, an opaque revocable server-side session, and an `HttpOnly`, `SameSite=Strict` cookie. Production cookies also require `Secure` and HTTPS.
- Browser mutations validate the expected `Origin`. Never accept identity, tenant, or role from request bodies or unverified headers.
- Services authorize every tenant-owned resource, including nested IDs, before data access or mutation.
- Reviewer/admin endpoints require explicit permissions and emit audit events.
- Return `404` rather than confirming existence when a resource belongs to another tenant.
- Email verification, password recovery, throttling, MFA, and optional OIDC federation are production launch work; adding them must preserve the service-boundary actor contract.

Implemented endpoints are registration, login, current-session lookup, logout, liveness, readiness, and authenticated SSE chat. Other resources in this contract describe the intended versioned API and must not be presented as currently available.

## 4. Success responses

- `200 OK`: successful read or synchronous update.
- `201 Created`: resource created; include `Location`.
- `202 Accepted`: asynchronous operation accepted; return a job resource/status URL.
- `204 No Content`: successful deletion/action with no representation.

Single-resource responses return the resource directly. Collections use:

```json
{
  "items": [],
  "page": {
    "next_cursor": null,
    "has_more": false
  }
}
```

Cursor pagination is preferred for changing collections. Cursors are opaque, signed or integrity-protected, scoped to normalized filters, and size-limited. Stable administrative reports may use documented page/limit pagination.

## 5. Error contract

Use `application/problem+json`, based on RFC 9457, with safe extensions:

```json
{
  "type": "https://api.msme-saarthi.example/problems/validation-error",
  "title": "Request validation failed",
  "status": 422,
  "detail": "One or more fields are invalid.",
  "instance": "/api/v1/enterprise-profiles",
  "code": "validation_error",
  "correlation_id": "2c2cbb42-0fe1-4e16-8bed-3c78e1719a03",
  "errors": [
    {"field": "facts.state_code", "code": "invalid_enum", "message": "Select a supported state code."}
  ]
}
```

`type`, `code`, and field error codes are stable machine interfaces. `title`, `detail`, and `message` are user-facing and may evolve/localize. Never include stack traces, SQL, prompts, secrets, provider payloads, or cross-tenant identifiers.

Standard statuses:

| Status | Use |
|---|---|
| `400` | Malformed semantics not attributable to a typed field |
| `401` | Missing/invalid authentication |
| `403` | Authenticated but action forbidden where existence disclosure is safe |
| `404` | Missing or concealed resource |
| `409` | State/version/idempotency conflict |
| `412` | Failed explicit precondition such as `If-Match` |
| `413` | Body exceeds endpoint limit |
| `415` | Unsupported media type |
| `422` | Schema/field validation failure |
| `429` | Rate limited; include `Retry-After` |
| `500` | Unexpected internal error |
| `502`/`503`/`504` | Dependency unavailable/overloaded/timed out |

## 6. Filtering, sorting, and search

- Query parameters are allowlisted and typed; reject unsupported filters rather than ignoring them.
- Repeat query keys for multi-value filters unless OpenAPI specifies another encoding.
- Sort values use `field` or `-field`; only indexed stable fields are allowed.
- Free-text queries are length-limited and normalized. They never become raw SQL/OpenSearch query DSL.
- Every list endpoint defines a deterministic tie-break sort.
- Publication and tenant filters are server-enforced, not client-selectable security controls.

## 7. Mutation safety

- `POST` creates or starts operations; non-idempotent create/operation endpoints accept `Idempotency-Key`.
- Idempotency records are scoped to actor, tenant, endpoint, and canonical request hash, with a documented expiry. Reuse with a different body returns `409`.
- `PUT` replaces only where supported; `PATCH` uses JSON Merge Patch (`application/merge-patch+json`) and rejects unknown fields.
- Mutable resources expose an ETag/version; updates accept `If-Match` to prevent lost updates.
- Historical profile versions, scheme versions, source versions, rule-set versions, and completed assessments are not patched.

## 8. Eligibility contract

Eligibility endpoints invoke only the deterministic engine. A completed assessment includes:

```json
{
  "id": "uuid",
  "status": "completed",
  "decision": "insufficient_information",
  "profile_version_id": "uuid",
  "scheme_version_id": "uuid",
  "rule_set_version_id": "uuid",
  "engine_version": "string",
  "missing_fact_paths": ["facts.annual_turnover"],
  "rule_outcomes": [],
  "citations": []
}
```

- Decisions are only `eligible`, `ineligible`, or `insufficient_information`.
- `eligible` means the supplied facts satisfy the encoded rules; it is not approval.
- Explanatory prose is separately identified and cannot alter `decision` or rule outcomes.
- Infrastructure/invalid-rule failure returns a failed operation/error, never `ineligible`.

## 9. Citations and AI responses

- A citation includes an opaque `citation_id`, source label, approved URL, source version, captured/verified time, locator, and bounded excerpt where permitted.
- Assistant responses use structured claim markers that resolve to citations returned in the same response.
- A material claim without resolvable support is removed, qualified as unsupported, or causes a safe fallback.
- Retrieved text and user content are untrusted data. APIs never return hidden prompts, reasoning, or internal safety policies.
- Streaming uses Server-Sent Events for one-way assistant output unless an ADR selects another protocol. Event schemas are versioned; a final event contains authoritative citations and completion status.

## 10. Asynchronous jobs

`202` responses return:

```json
{
  "job_id": "uuid",
  "status": "queued",
  "status_url": "/api/v1/jobs/uuid"
}
```

Job states are `queued`, `running`, `succeeded`, `failed`, and `cancelled` where cancellation is safe. Poll endpoints support `Retry-After`. Failures expose stable sanitized codes, not worker traces. Job submission and handlers are idempotent.

## 11. Observability and limits

- Accept a valid `X-Request-ID` or create one; return it and propagate as the correlation ID.
- Trace context follows W3C Trace Context. Do not expose internal topology in response headers.
- Apply per-route body, field, page, timeout, concurrency, and rate limits.
- Log route templates, status, latency, actor/tenant pseudonymous identifiers, and correlation ID; exclude tokens, prompts, raw facts, and response bodies.
- Define dependency timeout budgets shorter than the inbound request timeout.

## 12. Contract lifecycle

- Commit OpenAPI snapshots/generated client changes with the implementation.
- CI detects unreviewed breaking changes.
- Deprecations are documented and signaled with standard headers where practical.
- Contract tests cover examples, errors, auth, pagination, idempotency, version conflicts, and unknown fields.
- Production supports only explicitly documented API versions; removal requires usage review and migration notice.
